# 智能指挥车 AI 大脑产品需求文档

版本：v0.1  
日期：2026-05-11  
适用项目：公安应急处突指挥车系统、三维可视化移动指挥平台、刑事案件现场生成式实景 AI 三维重建及应用系统  
文档性质：产品需求文档 PRD  

## 1. 背景与定位

本项目的最终产品形态不是单一的 3DGS 建模工具，也不是普通的视频监控平台，而是装载在公安应急处突指挥车和相关移动指挥硬件之上的“AI 大脑”。

硬件系统负责抵达现场、采集现场、保障通信、提供算力和显示环境；AI 大脑负责把多源现场信息转化为可理解、可研判、可调度、可复盘、可举证的行动态势。

产品面向以下场景：

1. 突发公共安全事件现场处置。
2. 大型活动安保和大客流现场保障。
3. 群体性聚集风险感知与现场秩序维护。
4. 刑事案件现场快速三维重建、证据固定和案情分析。
5. 灾害、事故、复杂地形现场的移动指挥和事后复盘。

本产品应成为指挥车硬件体系的智能中枢，连接无人机、车载摄像机、复眼光学设备、亿级数字化成像设备、雷达、350M/4G/5G/卫星通信、车载算力、公安业务系统和后方指挥中心。

## 2. 产品目标

### 2.1 总目标

建设一套运行在指挥车本地并可与后方中心协同的智能指挥平台，实现现场数据快速接入、AI 态势感知、三维实景重建、重点线索发现、风险预警、指挥调度、证据固化和事后复盘。

产品应让指挥员在突发现场获得四类能力：

1. 看得全：多源视频、无人机、车载设备和三维场景形成统一现场态势。
2. 看得懂：AI 对人、车、物、区域、行为、风险、轨迹进行结构化分析。
3. 指得准：系统辅助生成处置建议、广播任务、警力调度和重点区域管控方案。
4. 查得回：所有视频、模型、识别结果、人工复核、指令和处置过程可追溯、可复盘、可举证。

### 2.2 阶段目标

第一阶段目标是形成“可指挥、可感知、可留证”的智能指挥车大脑：

1. 多路现场视频和无人机接入。
2. 现场人、车、物和基础风险识别。
3. 人群密度、聚集、拥堵、奔跑、推搡、跌倒、烟火、障碍物等风险提示。
4. 重点区域标注、事件时间线、原始视频留存。
5. 基础三维态势展示和静态 3DGS 建模能力。
6. 一键广播、分组广播、任务下发和回执。

第二阶段目标是形成“三维态势底座 + 重点目标智能分析”：

1. 重点区域预建三维底图。
2. 无人机位置、视锥、航迹和目标轨迹叠加。
3. 亿级数字化成像和授权人脸候选比对。
4. 行为识别、目标跟踪、轨迹分析和重点区域风险研判。
5. 与公安业务系统进行受控数据交换。

第三阶段目标是形成“4D 复盘 + 证据链 + 辅助决策闭环”：

1. 按时间轴回放现场三维态势、视频证据、目标轨迹和指挥指令。
2. 支持刑事案件现场、重点物证、路径、伤痕、工具、尸检过程等三维重建和关联分析。
3. 支持自动生成复盘报告、证据材料包和处置过程记录。
4. 支持模型持续迭代、事件库沉淀和跨事件模式分析。

## 3. 非目标与边界

为保证产品可交付和使用安全，以下内容不作为第一阶段承诺：

1. 不承诺陌生现场 1 分钟内完成像素级全动态 4D 重建。
2. 不承诺单靠高空无人机稳定识别所有人脸、手部动作和细微行为。
3. 不承诺 3DGS 模型替代原始视频、勘验记录或人工确认成为唯一证据。
4. 不承诺 AI 自动作出处置决定、强制措施决定或法律结论。
5. 不承诺在无授权名单、无合规接入和无人工复核的情况下进行身份确认。
6. 不承诺完全依赖视觉模型判断建筑坍塌、爆炸、火灾蔓延等高风险专业结论。

系统定位是辅助指挥和辅助取证。所有身份、风险等级、处置建议和证据结论均应保留置信度、来源、时间戳、模型版本和人工复核记录。

## 4. 用户与角色

### 4.1 指挥员

职责：

1. 查看现场态势。
2. 审核系统风险提示。
3. 下达广播、调度、管控和处置指令。
4. 查看关键人员、关键区域和事件进展。
5. 发起复盘和报告生成。

核心诉求：

1. 快速掌握现场全局。
2. 快速定位风险点和重点区域。
3. 快速形成处置方案。
4. 确保指令可下达、可回执、可追踪。

### 4.2 情报分析员

职责：

1. 查看 AI 识别结果。
2. 复核重点目标候选、轨迹和行为。
3. 关联历史数据、授权名单和业务系统信息。
4. 输出情报摘要和研判意见。

核心诉求：

1. 降低人工搜索视频的工作量。
2. 快速聚合人、车、物、区域和时间线线索。
3. 保证分析过程可追溯。

### 4.3 无人机操控员

职责：

1. 操控无人机采集现场视频、照片和位姿数据。
2. 按系统提示补充重点区域视角。
3. 执行指挥员下发的侦察任务。

核心诉求：

1. 明确飞行任务和视角要求。
2. 查看无人机状态、视频质量和采集覆盖度。
3. 避免重复采集和盲区遗漏。

### 4.4 通信保障员

职责：

1. 管理 350M、4G/5G、卫星、专网、Mesh 等通信链路。
2. 保障指挥车与前端人员、后方中心的通信畅通。
3. 处理断网、弱网和链路切换。

核心诉求：

1. 实时看到链路状态。
2. 明确关键业务的通信优先级。
3. 在断网情况下保证本地核心功能可运行。

### 4.5 刑侦勘查人员

职责：

1. 采集案件现场照片、视频、无人机影像和物证数据。
2. 使用系统生成三维现场、物证模型和路径还原。
3. 标注关键证据并导出材料。

核心诉求：

1. 快速固定现场原始状态。
2. 降低专业设备依赖和培训成本。
3. 支持案件分析、审查和法庭展示。

### 4.6 后方指挥中心用户

职责：

1. 接收指挥车回传的态势、视频、模型和报告。
2. 进行跨区域、跨部门联动指挥。
3. 查看历史事件归档和复盘结果。

核心诉求：

1. 远程掌握现场。
2. 获取可信的结构化信息。
3. 与前方指挥车保持一致的事件视图。

## 5. 核心业务场景

### 5.1 突发聚集事件现场处置

事件发生后，指挥车抵达现场，快速展开无人机、车载摄像机、复眼光学设备和通信链路。系统接入多路视频并自动生成现场态势。

系统应识别：

1. 人群密度和聚集区域。
2. 人员流动方向和拥堵点。
3. 奔跑、推搡、跌倒、打斗、翻越、疑似持物等行为。
4. 重点区域、危险区域和通道阻断情况。
5. 授权范围内的重点人员候选结果。

系统应支持：

1. 指挥员在二维地图和三维态势上查看现场。
2. 对重点区域放大查看、标注和下发侦察任务。
3. 一键广播、分组广播和任务下发。
4. 记录全部视频、识别结果、人工复核和指令流。

### 5.2 大型活动安保

在大型活动前，系统加载重点区域三维底图和预案。活动中，系统接入无人机、固定摄像机和现场人员设备，持续分析人流、车流、拥堵和异常行为。

系统应支持：

1. 重点区域预案管理。
2. 人群密度热力图。
3. 出入口、通道、隔离栏和临时设施状态监测。
4. 重点风险自动提示。
5. 事件结束后的客流复盘和保障报告。

### 5.3 刑事案件现场重建

现场人员使用手机、相机、无人机等常规设备采集现场图片或视频。系统自动完成数据预处理、相机位姿估计、三维重建和模型展示。

系统应支持：

1. 现场所在区域地理位置和环境三维再现。
2. 中心现场三维重建。
3. 嫌疑人出入路径重建。
4. 犯罪工具、物证、客体和痕迹三维重现。
5. 尸体解剖过程三维重现。
6. 关键证据标注、量测、关联分析和材料导出。

### 5.4 灾害事故应急指挥

系统接入无人机、传感器、车载摄像机和人员回传信息，识别烟火、积水、障碍物、道路阻断、建筑损伤等风险。

系统应支持：

1. 灾害现场快速态势图。
2. 危险区域和救援通道标注。
3. 救援人员定位和任务回执。
4. 断网断电情况下本地自治运行。
5. 事后过程复盘和处置评估。

## 6. 产品总体架构

### 6.1 分层架构

系统采用“采集接入层、边缘计算层、AI 感知层、空间智能层、指挥调度层、证据复盘层、中心协同层”的架构。

```text
----------------------+------------------------------------------+
| 层级                 | 主要职责                                 |
+----------------------+------------------------------------------+
| 采集接入层           | 接入无人机、车载摄像机、复眼光学、雷达、 |
|                      | 手机、执法记录仪、传感器和通信设备       |
| 边缘计算层           | 视频解码、抽帧、时间同步、存储、GPU 推理 |
| AI 感知层            | 目标检测、跟踪、行为识别、人脸候选比对、 |
|                      | 风险识别和事件生成                       |
| 空间智能层           | 3DGS 重建、三维底图、轨迹投影、4D 回放   |
| 指挥调度层           | 告警、广播、任务下发、回执、处置建议     |
| 证据复盘层           | 原始视频留存、哈希、审计、报告和材料包   |
| 中心协同层           | 与后方指挥中心、公安业务系统和归档平台同步 |
+----------------------+------------------------------------------+
```

### 6.2 AI 大脑能力模型

AI 大脑由五个核心能力构成：

1. 感知：把视频、图像、雷达、定位和通信数据转化为结构化对象。
2. 理解：识别人、车、物、区域、行为、轨迹、风险和事件。
3. 研判：基于规则、模型、历史事件和现场态势生成风险等级与处置建议。
4. 调度：把研判结果转化为广播、任务、侦察路径、人员调度和资源分配建议。
5. 记忆：把原始证据、识别结果、人工复核、指令和处置结果沉淀为可追溯事件档案。

## 7. 功能需求

### 7.1 多源设备接入

系统应支持接入以下设备和数据源：

1. 多旋翼无人机实时视频。
2. 无人机照片、视频和飞行遥测数据。
3. 车载云台摄像机。
4. 复眼光学设备。
5. 亿级数字化成像设备。
6. 雷达或低空目标探测设备。
7. 无人机反制设备状态。
8. 350M 集群通信设备。
9. 4G/5G 聚合通信设备。
10. 卫星通信链路。
11. 现场警务人员手机、执法记录仪和定位设备。
12. 固定摄像机、ONVIF/RTSP 视频源。
13. 环境传感器和应急设备状态。

设备接入应满足：

1. 每个设备具有唯一编号、类型、位置、状态和数据源地址。
2. 视频流应带时间戳。
3. 无人机应上报 GPS/RTK 坐标、高度、速度、航向、云台角度、相机参数、电量和视频流地址。
4. 设备离线、弱信号、低电量和异常状态应触发提示。
5. 所有接入、断开、配置变更应写入审计记录。

### 7.2 视频与图像管理

系统应支持：

1. 多路视频预览。
2. 视频墙显示。
3. 指挥大屏显示。
4. 低延迟视频预览。
5. 视频录制和回放。
6. 关键帧自动抽取。
7. 视频质量检测。
8. 视频源按事件、设备、时间、区域检索。
9. 视频片段导出。
10. 原始视频哈希校验。

视频管理要求：

1. 原始视频不得被 AI 处理结果覆盖。
2. AI 推理应引用原始视频帧 ID 和时间戳。
3. 关键事件应能反查到原始视频片段。
4. 弱网时应优先保证语音、告警和低码率预览。
5. 网络恢复后应自动补传事件元数据和必要视频片段。

### 7.3 AI 目标识别

系统应识别和跟踪以下目标类型：

1. 人。
2. 车辆。
3. 无人机和低空目标。
4. 障碍物。
5. 临时设施。
6. 疑似危险物。
7. 烟火。
8. 积水。
9. 道路阻断。
10. 建筑明显损伤特征。

目标识别结果应包含：

1. 目标 ID。
2. 目标类别。
3. 置信度。
4. 视频源。
5. 时间戳。
6. 图像框或分割区域。
7. 所属区域。
8. 轨迹片段。
9. 模型版本。
10. 人工复核状态。

### 7.4 人群态势分析

系统应支持：

1. 人群密度估计。
2. 聚集区域识别。
3. 拥堵区域识别。
4. 人流方向分析。
5. 逆向流动提示。
6. 通道容量风险提示。
7. 踩踏风险等级提示。
8. 重点区域人数变化趋势。

人群态势输出应包含：

1. 区域名称或坐标范围。
2. 估计人数或密度等级。
3. 风险等级。
4. 趋势方向。
5. 触发原因。
6. 建议处置动作。
7. 关联视频和关键帧。

### 7.5 行为识别

系统应优先支持以下行为识别：

1. 奔跑。
2. 跌倒。
3. 聚集。
4. 推搡。
5. 打斗。
6. 翻越护栏。
7. 疑似持物。
8. 疑似非法拍摄或异常拍摄。
9. 长时间滞留。
10. 快速分散。
11. 目标尾随。

行为识别要求：

1. 输出置信度，不直接输出定性结论。
2. 低置信度结果进入待复核队列。
3. 关键行为应关联连续视频片段，而非单帧孤立判断。
4. 同一行为事件应支持多视角证据合并。
5. 误报、漏报和人工修正结果应回流模型评估系统。

### 7.6 授权人脸候选比对

在具备合法授权、合规数据源和图像质量满足要求的前提下，系统可支持人脸候选提取与比对。

功能要求：

1. 从亿级数字化成像系统、车载视频、无人机变焦画面和固定摄像头中提取人脸候选。
2. 对人脸质量进行评分。
3. 低质量人脸不得进入正式比对，只可记录为质量不足。
4. 与授权名单或授权业务库进行候选匹配。
5. 输出候选列表、相似度、置信度和来源。
6. 身份确认必须由授权人员人工复核。
7. 所有查询、比对、查看和导出操作必须审计。

限制要求：

1. 系统不得把候选比对结果自动标记为确定身份。
2. 系统不得在无授权名单和无合规依据的情况下启用身份比对。
3. 系统应支持按任务、时间、人员权限控制比对能力。

### 7.7 重点目标跟踪与轨迹分析

系统应支持：

1. 从视频中选择目标并持续跟踪。
2. 跨摄像头、跨无人机的目标轨迹关联。
3. 在二维地图和三维场景中显示轨迹。
4. 记录目标进入、离开、停留、接近、绕行等事件。
5. 对重点目标生成轨迹摘要。
6. 对轨迹关联结果进行人工确认。

轨迹数据应包含：

1. 目标匿名 ID。
2. 轨迹点。
3. 时间戳。
4. 数据源。
5. 置信度。
6. 关联事件。
7. 人工复核状态。

### 7.8 三维实景重建

系统应支持两类三维能力：

第一类是现场处置阶段的三维态势：

1. 加载预建三维底图。
2. 加载无人机实时位置、航迹和视锥。
3. 将目标、轨迹、风险区域和指挥标注叠加到地图或三维场景。
4. 对局部变化区域进行标注或快速重建。

第二类是事后复盘和刑侦分析阶段的高质量重建：

1. 使用照片或视频生成 3DGS 场景。
2. 支持无人机视频抽帧建模。
3. 支持手机、相机、无人机等常规设备采集数据。
4. 支持中心现场、出入路径、物证、工具、客体、痕迹和解剖过程的三维重建。
5. 支持模型浏览、量测、标注、裁剪和导出。

三维重建要求：

1. 原始素材必须留存。
2. 建模过程必须记录处理命令、参数、模型版本和日志。
3. 模型不得替代原始素材作为唯一证据。
4. 模型应能反查到对应图片、视频帧和采集时间。
5. 对动态人群场景，应区分静态背景、动态目标和轨迹标注，避免强行把动态对象融入静态 3DGS 背景。

### 7.9 亿级数字化成像系统接入

系统应接入亿级数字化成像设备，实现大范围、远距离、高分辨率现场观测。

功能要求：

1. 支持超大场景图像采集。
2. 支持局部区域放大查看。
3. 支持高分辨率图像中的人脸候选、人员、车辆和异常区域提取。
4. 支持图像与三维态势或二维地图对齐。
5. 支持重点区域截图、标注和归档。

### 7.10 雷达与低空目标管控

系统应支持低空目标发现和管控相关能力：

1. 接入雷达或低空探测设备状态。
2. 显示空中目标方位、高度、速度和轨迹。
3. 识别合作目标和非合作目标。
4. 与无人机任务和反制设备状态联动展示。
5. 对低空风险进行告警。

系统不得自动执行反制动作。反制建议、设备操作和处置动作必须由授权人员确认。

### 7.11 风险研判

系统应建立风险引擎，对以下风险进行分级提示：

1. 人群过密。
2. 拥堵。
3. 踩踏风险。
4. 打斗或冲突风险。
5. 重点目标异常移动。
6. 烟火。
7. 积水。
8. 道路阻断。
9. 建筑明显损伤。
10. 低空目标风险。
11. 通信链路异常。
12. 设备电量或算力异常。

风险引擎输出应包含：

1. 风险类型。
2. 风险等级。
3. 触发规则或模型。
4. 证据来源。
5. 置信度。
6. 影响区域。
7. 推荐处置动作。
8. 人工确认状态。

### 7.12 指挥调度

系统应支持：

1. 一键广播。
2. 分组广播。
3. 对单个现场人员发送任务。

8. 任务状态跟踪。

任务对象应包含：

### 7.13 证据链与审计

系统应把“原始证据可追溯”作为底层原则。

必须留存：

1. 原始视频。
2. 原始图片。
3. 关键帧。
4. 设备遥测。
5. 三维模型。
6. AI 识别结果。
7. 人工复核记录。
8. 指挥指令。
9. 任务回执。
10. 报告导出记录。
11. 文件哈希。
12. 模型版本。
13. 操作审计日志。

证据链要求：

1. 任一事件都能反查原始视频片段。
2. 任一 AI 结论都能反查模型版本、输入帧和置信度。
3. 任一人工修改都必须记录修改前后内容、人员、时间和原因。
4. 任一导出材料都必须有导出人、时间、范围和哈希。
5. 数据删除、脱敏、归档必须有审批或权限控制。

9. 自动生成复盘报告。
10. 导出证据材料包。

复盘报告应包含：

1. 事件基本信息。

10. 待改进事项。

## 11. 非功能需求

### 11.1 性能

第一阶段建议指标：

1. 支持不少于 4 路无人机或现场视频并发接入。
2. 局域网内视频预览延迟控制在 1 至 3 秒。
3. 关键帧级 AI 检测延迟控制在 1 秒左右。
4. 告警从生成到界面显示不超过 2 秒。
5. 指挥广播指令下发延迟优先保障。
6. 静态 3DGS 建模作为后台任务运行，不阻塞实时指挥。

第二阶段建议指标：

1. 支持 8 至 16 路视频接入。
2. 支持重点区域三维底图 10 秒内加载并可交互。
3. 支持无人机位置和视锥 1Hz 以上刷新，推荐 5Hz。
4. 支持关键目标轨迹秒级更新。

### 11.2 可靠性

系统应支持：

1. 指挥车本地自治运行。
2. 与中心断网时继续录制、识别、告警和广播。
3. 网络恢复后自动同步。
4. 关键服务异常自动重启或降级。
5. 存储不足提前预警。
6. GPU 任务排队和互斥，避免多个重建任务抢占算力。

### 11.3 安全与合规

系统必须满足：

1. 用户分级授权。
2. 角色访问控制。
3. 操作审计。
4. 数据加密存储。
5. 传输加密。
6. 敏感数据脱敏。
7. 人脸候选比对授权控制。
8. 证据数据防篡改。
9. 数据留存策略。
10. 导出审批或权限控制。

### 11.4 可维护性

系统应支持：

3. 设备配置。

## 13. 验收标准

### 13.1 现场指挥验收

8. 任务可回执。

1. 可导入照片或视频素材。

3. 人工可复核并修正结果。
4. 误报和漏报可记录。
5. 风险告警可分级。

### 13.4 证据链验收

1. 原始视频和图片保留哈希。
2. 关键帧和导出材料可追溯。

4. AI 结果和人工复核可追溯。

## 14. 主要风险与对策

| 风险 | 影响 | 对策 |
| --- | --- | --- |
| 现场视频质量差 | AI 识别和建模失败 | 建立质量检测、采集指引和失败诊断 |
| 无人机位姿误差大 | 三维投影和轨迹不准 | 使用 RTK/IMU/云台参数校准和人工修正 |
| 动态人群影响 3DGS | 模型漂浮或失真 | 区分静态底座、动态目标和轨迹层 |
| 网络不稳定 | 视频中断和指挥延迟 | 本地自治、多链路聚合、断点续传 |
| AI 误报漏报 | 指挥误判 | 告警分级、置信度、人工复核和模型迭代 |
| 人脸比对合规风险 | 数据安全和法律风险 | 授权名单、权限控制、审计、人工确认 |
| 算力不足 | 延迟升高 | 任务排队、抽帧策略、多 GPU 扩展 |
| 证据链不完整 | 复盘和举证不足 | 原始数据留存、哈希、审计和反查机制 |
| 需求过度追求实时 4D | 交付延期 | 分阶段建设，先实时态势，后高质量复盘 |

5. 多场景复制部署。

## 16. 待确认问题

以下问题需要在 TR0 或 PRD 评审中确认：

1. 第一批硬件清单和接口协议。
2. 车载 GPU、存储、网络和显示设备配置。
3. 无人机品牌、遥测接口、视频协议和相机参数。
4. 亿级数字化成像设备接口。
5. 复眼光学、雷达、无人机反制设备接口。
6. 是否接入公安网及具体数据边界。
7. 人脸候选比对的授权范围和合规流程。
8. 第一阶段是否必须部署在完全离线环境。
9. 数据留存周期和归档策略。
10. 第一批试点场景：群体事件、大型活动、刑侦现场或灾害应急。
11. 验收演练的现场条件、样本视频和人员规模。
12. 后方指挥中心需要同步哪些数据。

## 17. 结论

本产品应以“硬件已经具备现场采集和通信能力，软件负责形成思考的大脑”为核心定位。

第一阶段不应把项目压缩成单纯的 3DGS 建模工具，也不应直接承诺全自动实时 4D 重建。正确的交付核心是：

```text
多源接入
  -> AI 态势感知
  -> 风险研判
  -> 指挥调度
  -> 原始证据留存
  -> 三维增强
  -> 复盘举证
```

3DGS、高斯光场、亿级成像、人脸候选比对、行为识别和大模型空间理解都应服务于同一个产品目标：让指挥车从“移动硬件平台”升级为“可感知、可研判、可指挥、可记忆、可复盘”的现场智能中枢。
