下面从“分支太耗时”开始，把刚才讨论完整整理成一版工程结论。

**1. 当前耗时问题的真实原因**
当时我们看了新重跑任务 `8ab3c5d055f041558397257724108e97`：

- `pano_window_00_45_e8_exhaustive_full`：
  - 45 张全景
  - 每张展开 8 个透视视角
  - 共 360 张透视图
  - 耗时约 10 分钟

- `pano_full_all_e8_sequential_full`：
  - 259 张 prepared 全景
  - 每张展开 8 个透视视角
  - 约 2072 张透视图
  - 展开、特征提取、matching 很快完成，但 COLMAP bundle adjustment 非常耗时

所以瓶颈不是“分支数量”本身，而是：

```text
全景图数量 × images_per_equirect
```

导致 COLMAP 处理的相机数量暴涨。

另外，这个任务原始资产是 260 张，prepared 进入重建的是 259 张。少的那一张是：

```text
IMG_20260510_123358_00_595.jpg
原因：duplicate_nonrepresentative
```

如果业务要求“原始全素材全部参与”，全景路径里 duplicate 不应该默认剔除，只应该标风险。

**2. e4 / e6 / e8 的含义和影响**
`e4/e6/e8` 指每张 360 全景被 Nerfstudio 拆成多少个透视子视角：

```text
e4 = 每张全景拆 4 张
e6 = 每张全景拆 6 张
e8 = 每张全景拆 8 张
```

以 259 张全景为例：

```text
e4 -> 约 1036 张透视图
e6 -> 约 1554 张透视图
e8 -> 约 2072 张透视图
```

它会影响三件事：

- COLMAP 注册稳定性：视角更多，理论上重叠更密。
- 训练视角覆盖：局部边角、转角、遮挡区域可能更好。
- 计算成本：图像数量越多，COLMAP BA 和后续训练都更重。

但结论不是“e8 一定最好”。因为这些子视角都来自同一张全景、同一个相机中心，它没有增加新的真实几何基线。很多情况下，`e6` 的质量接近 `e8`，但成本明显更低。

讨论后的判断：

```text
e4：适合快速验证，不适合作为最终高质量交付默认
e6：适合作为工程默认，高质量和成本比较平衡
e8：适合 HQ fallback、关键项目、局部增强，不适合默认全量首跑
```

最重要的原则是：

```text
全素材 e6 通常比局部素材 e8 更有意义
```

之前“只建出一个角”的问题，不是 e8 不够，而是素材覆盖范围被分支缩小了。

**3. 快速验证的意义**
快速验证不是为了交付模型，而是为了避免直接烧几十分钟甚至更久。

它要回答：

- 这批素材能不能整体注册起来
- 是不是全素材都被纳入
- 是否走了正确的 panorama pipeline
- equirect 参数是否正确
- 有没有又只跑局部素材
- 是否值得继续跑更贵的 e6/e8

之前的 45 张 probe 有问题，因为它只验证局部。更合理的快速验证应该是：

```text
full_all_e4
```

也就是：

```text
全部 pano 都参与
但每张 pano 只拆 4 个透视视角
```

这样它验证的是“整体路径是否成立”，不是局部片段是否成立。

**4. e6/e8 应该做成可配置项**
这个假设成立，而且应该做。

建议把它从 preset 里抽成 job 参数：

```json
{
  "images_per_equirect": 6,
  "registration_mode": "balanced",
  "fallback_to_higher_equirect": true,
  "include_duplicate_pano": true
}
```

推荐模式：

```text
fast      -> full_all_e4
balanced  -> full_all_e6
hq        -> full_all_e8
```

默认应该是：

```text
full_all_e6
```

e8 不默认首跑，只作为高成本质量兜底。

**5. 多个子任务、多个 PLY 的想法**
这个可以做，但不适合作为“提升建模质量”的主线。

多个 PLY 更适合解决：

- 分块加载
- 先显示粗模型，再逐步显示细节
- 大场景浏览体验
- LOD 分层加载

但它不天然提高质量，反而有风险：

- 多个 PLY 坐标系可能不完全一致
- 分块边界可能有接缝
- 重叠区域 splats 可能冲突
- 曝光、密度、尺度可能不一致
- viewer 可能更卡

更合理的结构是：

```text
一个全局 base model
+ 多个局部 detail model
```

全局模型负责整体空间一致性，局部模型负责关键区域高质量展示。

**6. 在当前 e6 基础上圈定关键区域，再补 e8**
这是我们讨论里最值得产品化的方向。

推荐路径：

```text
全素材 e6 建全局模型
→ 用户/系统圈定关键 ROI
→ 反查哪些 pano / 哪些视角覆盖 ROI
→ 只对这些关键素材跑 e8
→ 训练局部 detail model
→ viewer 中作为局部高精度 overlay / LOD 加载
```

这比全量 e8 更合理，因为成本集中花在关键区域。

**7. 用户怎么圈关键区域**
用户不应该在 360 原图里打标，因为这不直观。

正确方式是：

```text
用户在建好的 3D 模型里圈 ROI
```

ROI 可以是：

- 3D bounding box
- 球形区域
- 模型表面点选区域
- 当前视角框选后反投影成 3D ROI

保存成类似：

```json
{
  "roi_id": "roi_001",
  "type": "box",
  "center": [x, y, z],
  "size": [w, h, d],
  "margin": 1.5
}
```

**8. 3D ROI 怎么关联回素材**
桥梁是：

```text
3D ROI
→ 世界坐标
→ 已注册相机位姿 transforms.json
→ 判断哪些 pano / 展开视角看得到 ROI
→ 生成 ROI_e8_detail 训练集
```

系统要对每个已注册相机视角判断：

- ROI 是否在视锥内
- ROI 投影到图像上的面积
- 距离是否合适
- 视角角度是否合适
- 是否可能遮挡
- 图像质量如何
- 该视角来自哪张原始 360 pano

对全景素材还要通过 mapping 回溯：

```text
frame_00123.jpg
→ IMG_20260510_xxx.jpg
→ source pano image_id
```

然后输出 ROI 训练素材清单：

```json
{
  "roi_id": "roi_001",
  "selected_pano_count": 32,
  "selected_view_count": 148,
  "source_panos": [
    {
      "image_id": "img_000128",
      "reason": "high_roi_visibility",
      "roi_projected_area": 0.18,
      "view_angle_score": 0.82,
      "distance_score": 0.76
    }
  ]
}
```

**9. ROI e8 不应该一开始重新跑局部 COLMAP**
最稳的是先复用全局 e6 的位姿。

推荐：

```text
global_e6 COLMAP poses
→ ROI source selection
→ ROI_e8 images
→ reuse/global-align poses
→ train local detail splat
→ viewer overlay
```

这样局部 detail model 能和全局模型保持同一坐标系，避免漂移。

如果全局 e6 的 pose 本身在该区域错了，那 ROI e8 只能提升纹理，不能根治几何错误，需要局部 COLMAP refine 或补采。

**10. 增量更新的原则**
补采数据不能覆盖旧结果。

它应该是一个新的 material batch：

```json
{
  "material_id": "material_002",
  "parent_run_id": "scene_001_global_e6_v1",
  "capture_type": "supplemental",
  "target_roi": "roi_001",
  "purpose": "repair_hole_area"
}
```

然后创建增量 run：

```text
run_type = incremental_update
base_run_id = global_e6_v1
target_roi_id = roi_001
```

原始补采素材也必须永久保留，不覆盖旧素材。

**11. 增量更新的三种模式**
第一种，最推荐：

```text
Local Detail Overlay
global_e6_base.ply
+ roi_001_detail_update.ply
```

优点是快、风险小、可回滚。

第二种：

```text
Expanded Dataset Retrain
旧素材 + 补采素材
→ 重新训练完整 scene_v2
```

适合补采范围大，或者旧模型整体不足。

第三种：

```text
Checkpoint Continue
```

不建议作为正式主线。数据集变了之后继续训练不一定稳定，证据链也不好解释。

**12. 怎么判断现场是否发生过变化**
增量更新前必须做 change detection，不能默认补采和旧模型是同一现场状态。

分四层：

- 元数据层：采集时间、EXIF、设备、操作员、现场封存记录
- 图像层：新旧 anchor 图局部匹配、边缘、物体轮廓、文字、标识是否变化
- 3D 对齐层：补采图能否注册到旧坐标系，重投影误差是否异常
- 模型层：新旧局部模型是否出现物体新增/消失、表面偏移、重影、穿插

要区分：

```text
appearance_change：光照、曝光、阴影、噪声变化
geometry_change：物体位置、形状、数量、空间结构变化
semantic_change：物体类别、文字、标识、证据内容变化
```

规则：

- 只是光照/曝光变化：可以作为同一现场状态增量更新。
- 临时遮挡：标记动态/遮挡风险，人工确认。
- 证据物、设备、家具、门窗、破损等发生位置/形状/数量变化：不能静默合并，必须做新版本。
- 无法判定：默认不覆盖旧交付，生成疑似变化报告。

**13. 第一次是全景，后补是直拍照片**
这是现实中很可能发生的情况。

处理原则：

```text
全景素材负责全局空间底座
直拍照片负责局部细节补强
```

直拍照片不要硬转成全景，也不要走 `panorama_360_sequence`。它应该是：

```text
supplemental_detail_photos
```

或者：

```text
perspective_image_sequence
```

对应 run：

```text
base material: panorama_360_sequence
supplement material: perspective_image_sequence
run_type: incremental_update
target_roi: roi_001
```

流程：

```text
全景 e6 建全局模型
→ 用户圈 ROI
→ 导入补拍直拍照片
→ 直拍照片质检
→ 与 ROI 附近全景 anchor 视角匹配
→ 评估能否注册到旧坐标系
→ 可融合则训练局部 detail
→ 不可融合则作为 ROI 高清参考
```

**14. 直拍照片和全景大概率不好衔接**
你指出的这个问题是对的。

近距离直拍通常有这些问题：

- 焦距更近
- 视角更窄
- 缺少周边上下文
- 和全景展开视角尺度差异大
- 很难稳定匹配
- 强行融合容易重影、错位、尺度漂移

所以直拍照片默认不能作为“可融合素材”。

它应该先进入桥接能力评估：

- 能不能和全景 anchor 视角匹配到足够特征点
- 是否包含 ROI 周边固定参照物
- 重投影误差是否合理
- 相机位姿是否稳定
- 是否只是局部特写，没有上下文

**15. 直拍补采的三种交付模式**
根据桥接质量分成：

```text
reference_only
```

直拍图只挂在 3D ROI 上，点击查看高清细节，不参与几何融合。

```text
anchored_detail_overlay
```

能粗略对齐，可以做局部贴片或视角热点，但标记非严格几何融合。

```text
local_detail_model
```

只有补拍有足够过渡视角和上下文时，才训练局部 detail splat 并叠加。

质量等级可以对应：

```text
A/B：可做 local_detail_model
C：只能做 anchored overlay
D：reference_only，需要补拍桥接图
```

**16. 补采 SOP 必须加入“桥接照片”**
如果只拍近距离细节，大概率无法和全景衔接。

补采建议应该是：

```text
中景桥接图
→ 过渡图
→ 细节特写
→ 多角度补充
```

每个 ROI 推荐：

- 3-5 张中景图：包含 ROI 和周边固定参照物。
- 4-8 张过渡图：从全景能识别的位置逐步靠近，保持 60%-80% 重叠。
- 8-20 张细节图：拍清楚文字、破损、材质、边界。
- 至少 2-3 个角度：避免只有正面照片。
- 不用数码变焦。
- 尽量固定曝光和焦距。
- 避免反光、运动模糊。
- 如果需要尺度参照，必须记录它是否后放，不能混成原始现场事实。

系统应该在 ROI 选定后生成补采清单，比如：

```text
roi_001:
- 需要中景桥接图：5 张
- 需要过渡图：6 张
- 需要细节图：12 张
- 必须包含参照物：门框左边缘、墙面裂缝、设备铭牌
- 风险：当前全景该区域纹理弱，直拍若无上下文将只能 reference_only
```

**17. 最终形成的工程路线**
我们讨论后比较现实的路线是：

```text
全局默认：full_all_e6
快速验证：full_all_e4
高质量兜底：full_all_e8
关键区域增强：global_e6 + ROI_e8_detail
补采增量：ROI supplemental material batch
直拍补采：先 bridge/alignment 评估，再决定 reference_only / overlay / detail model
```

一句话总结这段讨论的核心结论：

```text
不要全量 e8 一把梭。
先用全素材 e4/e6 建稳定全局底座，
再把 e8、直拍补采、增量更新这些高成本能力集中用在关键 ROI。
能融合就融合，不能融合就作为带证据链的高清参考，不强行拼接。
```
