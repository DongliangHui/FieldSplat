从这套参数看，问题不太像“4090 算力不够”或“30k 不够”，而是你们的配置里有几个**会直接限制最终效果的关键点**。

## 当前参数的主要问题

### 1. 最大风险：用 RealBasicVSR 增强结果直接走静态 3DGS

你们的 hypothesis 里写的是：

```json
"Train the unchanged static PLY pipeline on the MMagic RealBasicVSR time-consistent output..."
```

这说明当前链路是：

```text
原始视频/图片
→ RealBasicVSR 时序增强
→ COLMAP
→ splatfacto-big 30k
```

这个链路有风险。

RealBasicVSR 适合做视觉增强，但它可能会带来：

| 风险                         | 对 3DGS 的影响                     |
| ---------------------------- | ---------------------------------- |
| 纹理被平滑                   | COLMAP 特征点变少，位姿更差        |
| 细节被 AI 补出来             | 3DGS 学到伪纹理、假结构            |
| 帧间时序稳定但单帧几何不真实 | 新视角下出现糊、浮点、重影         |
| 边缘锐化/超分伪影            | Gaussian 边缘脏、刺状、透明雾      |
| 动态物体被增强得更稳定       | 静态 3DGS 更容易把动态目标学进场景 |

**建议不要用 VSR 输出同时做 COLMAP 和训练。**

更稳的方式是双流：

```text
A. 几何流：原始素材 / 轻度去模糊 / 去畸变
   → COLMAP / hloc / MASt3R
   → 得到相机位姿和稀疏点云

B. 外观流：RealBasicVSR / 去噪 / 超分结果
   → 只用于 3DGS 训练外观
   → 不参与特征匹配和相机位姿求解
```

也就是：**位姿尽量用真实图像，外观可以用增强图像。**

------

### 2. `bilateral_grid: false` 不适合你们这种素材

你们现在是：

```json
"bilateral_grid": false
```

如果素材里有视频增强、曝光变化、不同设备、不同时间段、压缩噪声，建议打开。

推荐改成：

```json
"bilateral_grid": true
```

对应命令大概是：

```bash
ns-train splatfacto-big \
  --data <processed_data> \
  --max-num-iterations 30000 \
  --pipeline.model.use_bilateral_grid=True \
  --pipeline.model.use_scale_regularization=True
```

`bilateral_grid` 对这类问题比较有用：

```text
曝光不一致
白平衡不一致
视频增强带来的局部色彩变化
不同帧 ISP 风格不一致
压缩噪声和锐化差异
```

它不一定能解决几何问题，但能明显减少“脏、糊、浮、色块不一致”的问题。

------

### 3. `min_registered_ratio: 0.7` 太低

你们现在允许：

```json
"min_registered_ratio": 0.7
```

这意味着 300 帧里只要注册 210 帧就算通过。

对普通试验可以，但对你们想要“效果好”的工程模型来说，这个阈值偏低。**70% 注册率不代表 COLMAP 质量好，只代表勉强能跑通。**

建议分两级：

```json
"min_registered_ratio": 0.85
```

高质量模式建议：

```json
"min_registered_ratio": 0.9
```

但更重要的是，不要只看 registered ratio。还要看：

| 指标                               | 建议                   |
| ---------------------------------- | ---------------------- |
| 注册率                             | 最好 > 85%             |
| 稀疏点云覆盖                       | 主体区域必须完整       |
| 相机轨迹                           | 不能有跳变、翻转、漂移 |
| track length                       | 越长越稳               |
| 重投影误差                         | 异常帧要剔除           |
| 每张图的有效匹配点                 | 太少的帧要剔除         |
| 是否只有一个主 connected component | 不能分裂成多个相机簇   |

你们当前的 `min_registered_ratio: 0.7` 容易让“位姿不够好但勉强通过”的数据进入 3DGS，后面再怎么训也救不回来。

------

### 4. `frame_target: 300` 不一定比 150 好

你们现在：

```json
"frame_target": 300,
"process_frame_targets": [300, 179, 150]
```

这个思路是对的：多档帧数尝试。

但问题是：**帧数不是越多越好。**

视频抽帧里，300 帧可能带来大量冗余帧：

```text
视角差异太小
运动模糊
滚动快门
重复纹理
动态人车
压缩噪声
曝光跳变
```

这些帧会污染 COLMAP，也会污染 3DGS。

比固定 300 更好的策略是：

```text
先抽 500~1000 候选帧
→ 模糊检测
→ 曝光异常检测
→ 相似帧去重
→ 根据视角覆盖/运动基线选帧
→ 生成 150 / 220 / 300 三档
```

也就是不要：

```text
均匀抽 300 帧
```

而要：

```text
选 300 张最适合重建的关键帧
```

### 5. `full_res_from_start: false` 可以试改成 true

你们现在：

```json
"data_downscale_factor": 1,
"full_res_from_start": false
```

这有点矛盾。

`data_downscale_factor: 1` 表示不降采样，保留原始分辨率；但 `full_res_from_start: false` 说明训练初期可能不是从全分辨率开始。

如果你们追求细节，而不是速度，可以做一组对照：

```json
"full_res_from_start": true
```

尤其是建筑边缘、标识牌、设备、地面纹理这类细节，可能会有改善。

不过这不是最高优先级。它的优先级低于：

```text
位姿质量
bilateral_grid
mask
frame selection
VSR 双流
```

------

### 6. `min_ply_bytes` 不是有效质量指标

你们现在：

```json
"min_ply_bytes": 10485760
```

这个只能说明 PLY 文件不太小，不能说明质量好。

10MB 的 PLY 可能是：

```text
有效 Gaussian
也可能是浮点
也可能是动态物体残留
也可能是错误结构
也可能是雾状半透明垃圾点
```

建议新增真正的质量指标：

| 指标                      | 用途                       |
| ------------------------- | -------------------------- |
| train PSNR / SSIM / LPIPS | 看训练拟合                 |
| novel-view holdout PSNR   | 看新视角是否崩             |
| Gaussian 数量             | 判断是否异常膨胀           |
| opacity 分布              | 判断是否大量半透明浮点     |
| scale 分布                | 判断是否长条 Gaussian 失控 |
| rendered depth 稳定性     | 判断几何是否可信           |
| COLMAP reprojection error | 判断位姿质量               |
| 注册帧空间覆盖率          | 判断是不是只重建了局部     |

------

### 7. `nerfstudio:latest` 不建议用于工程实验

你们现在：

```json
"nerfstudio_image": "ghcr.io/nerfstudio-project/nerfstudio:latest"
```

这个不适合工程复现。

`latest` 每次拉取可能变，参数、默认值、gsplat 版本、CUDA 依赖都可能不同。你们会遇到一个问题：

```text
同一套参数，今天跑和下周跑，结果不一定完全一致。
```

建议固定版本，例如：

```json
"nerfstudio_image": "ghcr.io/nerfstudio-project/nerfstudio:<固定版本号>"
```

并记录：

```text
nerfstudio version
gsplat version
CUDA version
PyTorch version
COLMAP version
GPU driver
```

## 当前参数优先改动顺序

| 优先级 | 改动                                 | 重要性   |
| ------ | ------------------------------------ | -------- |
| 1      | RealBasicVSR 和 COLMAP 解耦          | 最高     |
| 2      | `bilateral_grid: true`               | 很高     |
| 3      | `min_registered_ratio` 提到 0.85/0.9 | 很高     |
| 4      | 加 `use_scale_regularization`        | 高       |
| 5      | 关键帧选择从“数量优先”改成“覆盖优先” | 高       |
| 6      | `full_res_from_start: true` 做对照   | 中       |
| 7      | `latest` 改固定版本                  | 工程必须 |
| 8      | 不再把 `min_ply_bytes` 当质量门槛    | 工程必须 |

## 我的判断

你们这套参数里，真正拖效果的最可能是这三点：

```text
1. RealBasicVSR 增强图直接影响了 COLMAP / 几何初始化
2. bilateral_grid=false，无法处理增强后的视频外观差异
3. min_registered_ratio=0.7，导致较差位姿也能通过
```

`splatfacto-big + 30k` 本身没错，但你们现在相当于把一个高质量训练器接在了一个不够稳的输入链路后面。下一步不要先换成别的 3DGS 模型，先把 **几何流、外观流、位姿验收、质量门槛** 这四件事改掉。
