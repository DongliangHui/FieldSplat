# 当前数据上传处理建模展示开发计划

状态：当前主执行计划  
日期：2026-05-12  
范围：数据抽取/上传、处理、建模、展示的浏览器端闭环  
基线：`browser-video-3dgs-production-plan-v1.0-20260511.md`、现有 `apps/api/video2splat` MVP

## 1. 目标

把当前离线 Video2Splat MVP 升级成“数据进入后可观察、可等待、可验收”的建模任务系统。

用户在浏览器创建任务后，必须立即进入任务详情页，并持续看到：

1. 上传/抽取、处理、建模、展示发布这几类阶段的实时进度。
2. 每个阶段始终无遮挡可见，不依赖弹窗、折叠区或只显示当前阶段。
3. 任务完成后，每个阶段的开始时间、结束时间、花费时间。
4. 最终产出物列表，包括类型、文件名、大小 MB、生成阶段、打开/查看入口。

本计划只保留现有视频上传到 3DGS 的链路。

## 3. 和原计划相比需要补齐的事

| 方向 | 原有基础 | 需要补齐 |
| --- | --- | --- |
| 数据入口 | 只支持单视频上传，创建 `VideoAsset` | 抽象成任务输入资产；短期保留视频，接口命名逐步过渡到 dataset/source asset |
| 阶段可见性 | 阶段运行时才插入 `pipeline_stage_runs` | 创建任务时预置完整阶段清单，未开始阶段也显示 |
| 实时刷新 | 3 秒轮询 | 增加 `/api/jobs/{id}/events` SSE；保留轮询兜底 |
| 阶段进度 | 只有 job 级 `progress`，阶段只有 running/succeeded/failed | 每阶段有 `progress`、`display_name`、`stage_order`、`duration_ms`、`output_summary` |
| 阶段耗时 | 已有 `started_at`、`finished_at`，未计算 | API 返回每阶段耗时；完成页显示阶段耗时和总耗时 |
| 产物大小 | artifact 有 `size_bytes`，页面显示为自适应单位 | 页面固定显示 MB；API 可直接返回 `size_mb` |
| 产物归属 | artifact 没有明确生成阶段 | artifact 增加 `stage` 或通过结果清单映射到生成阶段 |
| 处理阶段粒度 | `process_data` 同时包含抽帧、COLMAP、transforms | 拆成可观察的抽帧、质量预检、位姿/注册等阶段 |
| 展示发布 | 有 PLY Viewer | 增加 `publish_viewer` 阶段和 viewer 可用性检查 |
| 页面布局 | 详情页阶段列表、日志和产物同屏，但信息较粗 | 固定阶段看板 + 阶段结果表 + 产物表，核心信息不被日志遮挡 |

## 4. 新的产品链路

### 4.1 当前 P0 链路

```text
选择视频/素材
  -> 上传素材
  -> 创建建模任务
  -> 进入任务详情页
  -> 实时查看阶段进度
  -> 完成后查看阶段耗时
  -> 查看最终产物和 MB 大小
  -> 打开 3DGS Viewer
```

### 4.2 面向后续多源的目标链路

```text
SourceAsset / Dataset
  -> FrameAsset / Metadata / Quality
  -> Pose / Coverage / Registration
  -> 3DGS BaseLayer / Tile / Dynamic Layer
  -> SceneManifest
  -> Browser Viewer / Replay
```

P0 不要求一次性做完多源和流式，只要求当前单视频链路的任务可观察性达到产品级。

## 5. 页面要求

### 5.1 上传/创建页

页面职责：

| 要素 | 要求 |
| --- | --- |
| 文件选择 | 当前支持视频；控件文案使用“素材”而不是只写“视频”，为后续扩展预留 |
| 上传进度 | 上传时显示百分比、已上传 MB、总 MB；不能只显示“上传中” |
| 创建任务 | 上传成功后自动创建任务，失败时给出明确原因 |
| 跳转 | 创建任务成功后立即跳到 `/jobs/{jobId}` |
| 运行环境 | 保留 Docker/GPU/磁盘状态，但不遮挡上传主流程 |

实现建议：

1. 将上传从 `fetch(FormData)` 改成支持 progress event 的 `XMLHttpRequest`，或后续改 chunk upload。
2. 上传完成后调用现有 `/api/jobs`，短期保持兼容。
3. 长期新增 `/api/datasets`、`/api/assets`、`/api/reconstruction/jobs`。

### 5.2 任务详情页

这是本轮最重要的页面。

页面必须包含三块核心区域：

| 区域 | 要求 |
| --- | --- |
| 顶部任务摘要 | 任务状态、总进度、当前阶段、总耗时、输入素材大小 |
| 阶段看板 | 所有阶段从创建任务后就可见；每行显示状态、进度、开始/结束、耗时、主要输出 |
| 产物列表 | 完成或部分产出后显示文件名、类型、生成阶段、大小 MB、打开/查看入口 |

无遮挡要求：

1. 阶段看板不能放进默认折叠区。
2. 阶段看板不能被日志面板、浮层、toast 或 viewer iframe 遮住。
3. 日志可以在右侧或下方滚动，但不能覆盖阶段表。
4. 小屏幕下阶段看板优先显示，日志移到下方。
5. 单行内容过长时省略或换行，不能撑破布局。

阶段行建议字段：

| 字段 | 示例 |
| --- | --- |
| 阶段 | 抽取帧 |
| 状态 | 等待中 / 运行中 / 成功 / 失败 / 跳过 |
| 进度 | 0-100% |
| 开始时间 | `2026-05-12 14:03:20` |
| 结束时间 | `2026-05-12 14:04:11` |
| 耗时 | `51s` / `03m 12s` |
| 产出 | `frames: 100`、`transforms.json`、`splat.ply` |
| 大小 | `128.34 MB` |

### 5.3 Viewer 页

页面职责：

| 要素 | 要求 |
| --- | --- |
| 打开入口 | 只有主模型产物可用后才显示“查看” |
| 加载状态 | viewer 内显示模型加载耗时、splat 数量 |
| 产物信息 | 页面顶部或任务页显示 `splat.ply` 大小 MB |
| 失败兜底 | 浏览器加载失败时保留下载入口和错误原因 |

P0 继续使用当前 Spark 3DGS + PLY 方案。

## 9. 实时刷新方案

优先方案：SSE。

```text
Worker
  -> update db
  -> insert job_events
  -> /api/jobs/{id}/events emits event
  -> browser updates job summary / stage board / artifacts
```

前端策略：

1. 进入任务页后先 `GET /api/jobs/{id}` 获取完整快照。
2. 建立 `EventSource("/api/jobs/{id}/events")`。
3. 收到事件后局部更新页面。
4. SSE 断开时显示“实时连接已断开，使用轮询刷新”，并每 3 秒轮询。
5. 任务进入 `succeeded`、`failed`、`cancelled` 后关闭 SSE，但保留最终快照。

P0 可接受轮询兜底，但最终验收必须证明创建任务后页面会自动刷新，不需要手动刷新浏览器。

## 11. 页面验收标准

本轮页面验收以 `/jobs/{jobId}` 为中心。

| 验收项 | 标准 |
| --- | --- |
| 创建任务后实时刷新 | 创建任务后自动进入任务页；状态无需手动刷新即可变化 |
| 阶段无遮挡可见 | 任务页一次性显示完整阶段清单；日志、弹窗、iframe 不遮挡阶段看板 |
| 阶段状态完整 | 每阶段至少有 waiting/running/succeeded/failed/skipped |
| 阶段耗时 | 完成后每阶段显示耗时；未开始显示 `-` |
| 总耗时 | 任务完成后显示总耗时 |
| 最终产物 | 显示 `splat.ply`、`viewer-camera.json`、`result.json` 等产物 |
| 大小 MB | 产物大小固定以 MB 展示，保留两位小数 |
| 失败可理解 | 失败阶段高亮，显示错误码和日志入口 |
| 移动端 | 小屏幕下阶段看板仍完整可见，不横向遮挡 |

## 13. 不在本轮 P0 范围

1. 不在 P0 直接做完整实时 4DGS。
2. 不在 P0 强制替换 Nerfstudio。
3. 不在 P0 做多车、多任务并发 GPU 调度。
4. 不在 P0 做正式权限系统。
5. 不在 P0 做完整指挥车业务大屏，只做建模任务闭环和 Viewer 入口。

## 14. 推荐落地顺序

1. 先做阶段 manifest 和任务详情 API，使所有阶段在任务创建后可见。
2. 再做前端任务详情页，保证无遮挡、实时刷新、完成态摘要。
3. 然后补 artifact 的 MB 展示和阶段归属。
4. 接着拆解 `process_data` 的业务阶段和质量指标。
5. 最后再进入 dataset/source asset，多源和流式增量。

这个顺序能最大化复用现有 MVP，同时最快满足“创建任务后看到实时进度、阶段完整可见、完成后看到耗时和产物大小”的页面要求。

## 15. 面向低质量素材高质量建模的补充结论

本节是对当前计划的目标修正。前 1-14 节主要解决“上传、处理、建模、展示过程可观察、可等待、可验收”的产品闭环问题；但当前项目的真实目标已经升级为：

```text
尽可能允许现场低质量素材输入
  -> 自动诊断素材问题
  -> 自动选择增强、遮罩、位姿和建模策略
  -> 产出尽可能高质量、可缩放、细节清楚的 3DGS/3D 场景复现
```

因此，现有计划不能只停留在“任务成功生成 `splat.ply`”。后续必须补齐一条质量工程主线：

```text
输入质量诊断
  -> 时序一致视频增强
  -> 动态/低置信区域处理
  -> 稳健位姿和几何初始化
  -> 多分支重建对比
  -> 自动质量验收
  -> 只把通过质量门槛的分支发布到 Viewer
```

### 15.1 当前计划和高质量目标之间的差距

| 方向 | 当前计划覆盖 | 高质量目标下还缺什么 |
| --- | --- | --- |
| 任务可观察性 | 已规划阶段看板、耗时、产物、SSE | 还需要把质量指标、失败诊断、实验分支结果纳入同一套阶段和报告 |
| 输入质量 | 只规划 `quality_precheck` | 需要真实的 `quality_report.json`，覆盖模糊、压缩、曝光、动态区域、视差、覆盖度和位姿风险 |
| AI 增强 | 当前未作为主链路阶段 | 需要时序一致增强分支，不能默认使用单帧超分 |
| 动态内容 | 当前只依赖 Nerfstudio/3DGS 自身鲁棒性 | 需要动态区域 mask、低置信区域降权、语义/光流残差诊断 |
| 位姿求解 | 当前主要依赖 `ns-process-data video`/COLMAP | 需要 hloc、VGGT、MASt3R/DUSt3R 等 fallback 或候选分支 |
| 建模策略 | 当前主要是 `splatfacto`/`splatfacto-big` preset | 需要 raw/enhanced/masked/robust-pose 等多分支对比，不能盲跑一个 preset |
| 成功标准 | 当前偏向生成 artifact 和可打开 viewer | 需要 held-out view、PSNR/SSIM/LPIPS、局部 crop、viewer 截图和人工可读质量报告 |

### 15.2 质量诊断阶段必须升级

`quality_precheck` 不能只是“模糊/时长/分辨率/覆盖评分”的轻量展示，它必须成为后续 pipeline 的决策入口。

最小输出文件：

```text
F:\video2splat\jobs\<job_id>\reports\quality_report.json
```

建议包含：

| 指标组 | 需要记录的内容 | 用途 |
| --- | --- | --- |
| 基础视频 | duration、fps、width、height、frame_count、codec、color_space | 判断基础输入是否可用 |
| 清晰度 | Laplacian sharpness、运动模糊、低清晰帧比例 | 决定是否进入去模糊/增强分支 |
| 压缩和噪声 | block artifact、noise、banding、码率风险 | 决定是否进入去压缩/去噪分支 |
| 曝光和色彩 | 过曝、欠曝、曝光跳变、HDR/HLG/BT2020 | 决定是否做色彩空间归一化和曝光稳定 |
| 动态区域 | 人、车、树叶、草地、水面、阴影变化比例 | 决定 mask 或降权策略 |
| 几何覆盖 | 抽样帧视差、轨迹覆盖、环绕完整度 | 决定是否提示素材覆盖不足或切更强位姿路线 |
| 位姿风险 | COLMAP 注册率、重投影误差、轨迹断裂 | 决定是否切 hloc/VGGT/MASt3R fallback |

验收标准：

1. 每个 job 完成 `quality_precheck` 后必须有 `quality_report.json`。
2. 任务详情页必须显示质量摘要，而不是只显示“检查通过/失败”。
3. 后续 worker 分支选择必须引用质量报告中的字段。

### 15.3 AI 增强必须以时序一致为前提

已验证结论：单帧 Real-ESRGAN 类增强会带来涂抹、块状伪影和跨帧不一致纹理，不能默认接入主链路。

后续 AI 增强路线应以视频时序一致为前提：

| 优先级 | 路线 | 目标 |
| --- | --- | --- |
| P0 试验 | RealBasicVSR / BasicVSR++ / RVRT / MMagic 等视频增强 | 去压缩、去噪、轻量去模糊，同时保持跨帧一致 |
| P0 约束 | 只先跑 60-100 连续帧验证 | 控制成本，先看 temporal consistency 和 COLMAP/重建收益 |
| P1 接入 | 增强分支进入 worker | 只有超过 raw baseline 才允许成为可选 preset |
| 禁止项 | 默认单帧超分后直接训练 | 避免生成假纹理，破坏几何和 3DGS 训练 |

建议新增阶段：

| 顺序 | 阶段 key | 展示名 | 输出 |
| --- | --- | --- | --- |
| 52 | `temporal_enhance_probe` | 时序增强试验 | enhanced sample、consistency metrics |
| 54 | `enhancement_gate` | 增强质量判断 | accepted/rejected、原因 |

增强分支通过条件至少包括：

1. COLMAP 注册率不低于 raw baseline。
2. held-out LPIPS 不劣于 raw baseline。
3. 样张不能出现明显涂抹、块状伪影或凭空生成纹理。
4. 帧间一致性指标不能显著退化。

### 15.4 动态和低置信区域必须处理

当前真实素材的问题不只是“画面不够清楚”，还包括树叶、草地、阴影、人车等动态内容和低可信纹理。静态 3DGS 会把这些不一致信息平均掉，导致缩放后发糊。

建议新增：

| 能力 | 方法候选 | 输出 |
| --- | --- | --- |
| 动态检测 | 光流残差、帧差、轨迹一致性 | `dynamic_mask_report.json` |
| 语义 mask | SAM、SegFormer、Mask2Former 或轻量语义模型 | vegetation/person/vehicle/sky masks |
| 阴影/曝光变化检测 | 亮度变化、局部时序方差 | shadow/exposure masks |
| 训练降权 | mask loss、robust loss、低置信区域采样降权 | masked training branch |

建议新增阶段：

| 顺序 | 阶段 key | 展示名 | 输出 |
| --- | --- | --- | --- |
| 56 | `dynamic_mask` | 动态区域分析 | masks、dynamic ratio、mask report |
| 58 | `mask_gate` | 遮罩质量判断 | accepted/rejected、原因 |

验收标准：

1. 不能只输出 mask 文件，必须显示 mask 比例和主要风险区域。
2. mask 分支必须和 raw baseline 对比，不允许无指标直接替换主链路。
3. 如果动态区域比例过高，应在任务结果中标记“静态 3DGS 质量受限”。

### 15.5 位姿和几何初始化需要 fallback

当前链路主要依赖 `ns-process-data video` 中的 COLMAP。低质量现场素材下，只有 registered ratio 不足以说明位姿可靠。

需要补充：

| 能力 | 说明 |
| --- | --- |
| pose quality report | 记录 registered/planned、重投影误差、轨迹断裂、覆盖度 |
| hloc fallback | 当 COLMAP 弱纹理或重复纹理失败时尝试特征匹配增强 |
| VGGT/MASt3R/DUSt3R 评估 | 作为位姿/深度/点云初始化候选，不替代 3DGS 输出 |
| branch comparison | 同一输入下比较 COLMAP、hloc、VGGT/MASt3R 初始化结果 |

建议新增阶段：

| 顺序 | 阶段 key | 展示名 | 输出 |
| --- | --- | --- | --- |
| 62 | `pose_quality` | 位姿质量分析 | pose report、risk flags |
| 64 | `pose_fallback` | 位姿备用求解 | fallback transforms 或 rejected reason |

### 15.6 建模阶段要从单 preset 变成分支对比

为了追求高质量复现，worker 不应该只根据 `fast/standard/high` 跑一个固定路径。建议引入 reconstruction branches：

| 分支 | 输入 | 用途 |
| --- | --- | --- |
| `raw_baseline` | 原始抽帧 | 永远保留，作为对照组 |
| `temporal_enhanced` | 时序一致增强帧 | 验证增强是否真实提升 |
| `masked_raw` | 原始帧 + 动态/低置信 mask | 验证动态处理收益 |
| `enhanced_masked` | 增强帧 + mask | 作为高质量候选 |
| `robust_pose` | 更强位姿/几何初始化 | 解决 COLMAP 不稳问题 |

每个分支必须输出统一报告：

```text
branch_id
input_variant
pose_variant
registered_frames
training_method
iterations
splat_count
ply_size_mb
eval_psnr
eval_ssim
eval_lpips
viewer_url
accepted
reject_reason
```

只有 `accepted=true` 且质量指标超过 baseline 的分支，才能成为默认发布分支。

### 15.7 质量验收必须成为最终 gate

原计划的 P0 成功标准是“生成 `splat.ply` 并能打开 viewer”。对当前目标来说，这只能证明链路跑通，不能证明建模质量达标。

必须新增最终质量 gate：

| 验收项 | 标准 |
| --- | --- |
| held-out render | 从训练中留出若干帧，渲染后和原始帧对比 |
| 指标 | PSNR、SSIM、LPIPS 必须记录到 `quality_report` 或 `result.json` |
| 局部 crop | 建筑边缘、地面、植被、阴影区域分别出 crop 对比 |
| viewer 截图 | 自动保存首屏和几个固定视角截图 |
| 人可读结论 | 输出“成功 / 可用但低质 / 失败 / 静态 3DGS 不适合该素材” |
| 发布条件 | 未通过质量 gate 时，不能把任务标记成高质量成功 |

建议新增阶段：

| 顺序 | 阶段 key | 展示名 | 输出 |
| --- | --- | --- | --- |
| 112 | `render_eval` | 渲染质量评估 | metrics、comparison crops |
| 114 | `quality_gate` | 最终质量门槛 | accepted/rejected、quality level |

### 15.8 调整后的推荐优先级

在保留第 14 节原有落地顺序的基础上，面向“低质量素材高质量建模”的真实目标，优先级调整为：

1. 先补任务可观察性最小闭环：阶段 manifest、阶段耗时、产物 MB、失败原因、中文乱码修复。
2. 立刻把 `quality_precheck` 升级为真实 `quality_report.json`，让它成为 pipeline 决策入口。
3. 新增时序一致视频增强实验分支，只跑小样本验证，不默认替换 raw。
4. 新增动态/低置信区域 mask 分支，验证是否提升静态区域细节。
5. 新增 pose quality 和 fallback 验证，解决低质素材下 COLMAP 不稳定问题。
6. 新增多分支重建对比和质量 gate，不再只以 `splat.ply` 生成作为成功。
7. 质量闭环稳定前，不再引入非 MP4 输入路线。

### 15.9 阶段清单增补建议

为兼容现有 P0 阶段，同时支撑高质量建模目标，建议把阶段清单扩展为：

| 顺序 | 阶段 key | 展示名 | 说明 |
| --- | --- | --- | --- |
| 10 | `asset_upload` | 素材上传 | 原始素材入库 |
| 20 | `task_prepare` | 任务准备 | job 目录、输入拷贝 |
| 30 | `inspect_video` | 素材解析 | ffprobe、基础元数据 |
| 40 | `extract_frames` | 抽取帧 | images、frame count |
| 50 | `quality_precheck` | 输入质量诊断 | `quality_report.json` |
| 52 | `temporal_enhance_probe` | 时序增强试验 | 小样本增强和一致性指标 |
| 54 | `enhancement_gate` | 增强质量判断 | 是否允许增强分支继续 |
| 56 | `dynamic_mask` | 动态区域分析 | masks、dynamic ratio |
| 58 | `mask_gate` | 遮罩质量判断 | 是否允许 mask 分支继续 |
| 60 | `camera_solve` | 位姿求解 | transforms.json |
| 62 | `pose_quality` | 位姿质量分析 | pose report |
| 64 | `pose_fallback` | 位姿备用求解 | hloc/VGGT/MASt3R 候选 |
| 70 | `registration_gate` | 注册质量判断 | accepted/retry diagnostics |
| 80 | `train_splatfacto` | 高斯建模 | raw/enhanced/masked branches |
| 90 | `export_gaussian_splat` | 模型导出 | splat.ply |
| 100 | `export_cameras` | 相机导出 | viewer-camera.json |
| 110 | `publish_viewer` | 展示发布 | viewer URL、artifact manifest |
| 112 | `render_eval` | 渲染质量评估 | metrics、comparison crops |
| 114 | `quality_gate` | 最终质量门槛 | accepted/rejected、quality level |
| 120 | `final_report` | 结果汇总 | result.json、stage summary |

短期实现时，不要求一次性实现所有 AI 和 fallback 算法；但阶段、报告结构和质量 gate 要先预留。这样后续每次引入新模型或新分支，都能用同一套质量指标和任务页面验收，而不是靠主观观看判断。
